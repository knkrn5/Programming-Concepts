Hard links vs symlinks

Hard links: store → .pnpm (saves disk space, since content is shared across all projects on your machine)
Symlinks: .pnpm → your project's node_modules (builds the correct, non-flat dependency graph)


One nuance: hard links only work within the same filesystem/volume. If your store and project are on different drives, pnpm falls back to copying (or you'd configure package-import-method differently).

how migrate the existing project from npm to pnpm ?


Orchestrated comprehensive migration strategy encompassing dependencies, configuration, and CI updates
Orchestrated comprehensive migration strategy encompassing dependencies, configuration, and CI updates
Here's the clean migration path:

1. Navigate to your project and remove npm's artifacts:

bash
rm -rf node_modules
del package-lock.json
(On Windows PowerShell: Remove-Item -Recurse -Force node_modules and Remove-Item package-lock.json)

2. Let pnpm generate its lockfile:

You have two options here:

Option A — Fresh install (simpler, recommended for most cases):

bash
pnpm install
This reads your package.json and resolves everything fresh, generating a new pnpm-lock.yaml. Version ranges in package.json are respected, but exact resolved versions may shift slightly from what package-lock.json had (usually fine, but check if you have strict version pinning needs).

Option B — Import existing lockfile (preserves exact resolved versions):

bash
pnpm import
This reads your package-lock.json (if you haven't deleted it yet — so do this before step 1's del package-lock.json) and converts it directly into pnpm-lock.yaml, keeping the exact same resolved versions you had under npm. Then run:

bash
pnpm install
to finalize.

3. Check for phantom dependencies:

This is the part that actually matters. pnpm's strict node_modules structure means code that was secretly relying on hoisted transitive dependencies (working under npm/yarn by accident) will now break with "module not found" errors. Run your build/tests:

bash
pnpm run build
pnpm run test
Any missing-module errors here mean you need to add that package explicitly to your package.json — it was a phantom dependency you got away with before.

4. Update your scripts and CI:

Update package.json scripts if they call npm run internally — swap to pnpm run (or just pnpm <script>, no run needed for pnpm)
Update CI/CD pipelines (GitHub Actions, etc.) — replace npm ci with pnpm install --frozen-lockfile
If using Docker, update your Dockerfile's install step similarly
5. Add a .npmrc if you want stricter/looser behavior (optional):

# example: if you need some hoisting for legacy tooling
shamefully-hoist=false
6. Clean up:

bash
git add pnpm-lock.yaml
git rm package-lock.json
Commit both changes together so your team's next git pull triggers a clean pnpm install rather than a broken hybrid state.

One thing specific to your Print Campus setup: if your Next.js web frontend has any next.config.js transpile/webpack customizations that assumed npm's flat node_modules, double check those after migrating — pnpm's stricter linking occasionally surfaces resolution issues with certain Next.js plugins that reach into node_modules internals. Worth running pnpm run dev and clicking through a few pages before you fully commit to the switch.
