in the pnpm store i never see a package wiht it name: -
Instead, pnpm uses a content-addressable store. Files are stored according to a hash of their contents, so you usually see folders and files with names similar to:
files/
  00/
  01/
  a4/
  ff/     

Inside this store, it doesn't just name folders by package name. It uses content-based hashing.(hash of the content in that package) now even if two package have same content, their hash will always be same
This ensures that if you have two different versions of a library, or two different packages that share the same underlying files, they aren't duplicated.

the pnpm calculate the hash after both downloading the package and before downloading the package
Before downloading: pnpm receives the expected hash from the npm registry.
After downloading: pnpm calculates the tarball hash to verify integrity.
After extraction: pnpm calculates file hashes to organize the content-addressable store.

// Incase of reinstall pnpm: - 
When you run corepack enable (or corepack prepare pnpm@<version> --activate), it just installs the pnpm binary/shims. On first use, pnpm checks its default store location and will find your existing store there — no need to re-download packages you already have cached.
-----------------------------------------------------
1. >>> When pnpm resolves a package, the registry gives back metadata that already includes the package's integrity hash (dist.integrity, a SHA-512) — pnpm doesn't have to compute anything itself for this step. pnpm maintains a SQLite-backed index (index.db) to map package identifiers like tarball integrity to the list of files they contain. So pnpm just looks up that known hash against its local store index — if a match exists, it skips the download entirely and links straight from the store. This is the dedup check, and it happens before any bytes are fetched, because the hash is already known from metadata, not computed locally.

2, >>> Only if there's no match does pnpm actually fetch the tarball. Once downloaded, pnpm recomputes the hash on every install and verifies it matches the integrity field from the registry metadata. This is the actual authenticity/corruption check — confirming what arrived over the wire matches what the registry claimed it would send

Hard links vs symlinks

Hard links: store → .pnpm (saves disk space, since content is shared across all projects on your machine)
Symlinks: .pnpm → your project's node_modules (builds the correct, non-flat dependency graph)


One nuance: hard links only work within the same filesystem/volume. If your store and project are on different drives, pnpm falls back to copying (or you'd configure package-import-method differently).

pnpm dlx — pnpm's equivalent of npx. It downloads a package temporarily (not installed globally, not added to any package.json), runs it once, then discards the cached copy after use (or reuses pnpm's content-addressable store cache on later runs). It's the "run this CLI tool without permanently installing it" command.
pnpm dlx — the standard "download and execute" command (like npx).
pnpx — a shorthand alias for pnpm dlx, same behavior, just fewer keystrokes.
pnx — pnpm's own newer/canonical binary name for the same dlx functionality, with some extra flags like --package to pull in a specific package/version before running a different command, and support for security policy settings (minimumReleaseAge, etc.) as of pnpm v11.

// folders in pnpm store: - 
files — this is the one you're asking about. It's the actual content-addressable store where every package's file contents live, stored by content hash (not by package name/version). This is where the real disk space is used.
links — internal bookkeeping pnpm uses to track hard-link relationships between the store and your projects.
projects — metadata about which projects on your machine reference which packages from the store (this is how pnpm store prune knows what's safe to delete).
index — a database file (SQLite-based) that maps package names/versions to their content hashes in files, so pnpm can quickly resolve what needs to be linked without re-fetching.
>> : files is where all your actual packages (the code) are stored. The other three are pnpm's internal indexing/tracking system that makes the hard-linking mechanism work correctly.

Each of those long hex strings (5331b2bd8b60feef30e5bf5e90a328cb, etc.) is a hash of a project's absolute path on your disk. So:

pnpm computes a hash of a project directory's path (e.g. C:\Users\User\some-project)
Creates a folder here named with that hash
Inside it, keeps a record of exactly which packages/versions that specific project has installed and is linking to from the store
Why this exists: Since the actual package content lives once in files (shared across all projects), pnpm needs some way to know "is anyone still using this package?" before it can safely delete it. This projects folder is that reference-counting mechanism — it's what pnpm store prune reads to figure out which blobs in files are orphaned (no project references them anymore) versus still in use.
